# FitLog

**v2.0 要件仕様書**

トレーニング基盤（種目マスタ・セッション・セット記録）

2026年4月 / Version 2.0

---

## 1. 概要

v1.x で構築した体重・体脂肪率記録の基盤に、筋力トレーニングの記録機能を追加する。
種目マスタ（`exercises`）・セッション（`workout_sessions`）・セット（`workout_sets`）の3テーブル構成で、セッション単位のCRUDとトレーニング履歴の閲覧を実現する。

RPE・ボリューム推移・漸進性過負荷トラッキングは v2.1 / v2.2 で段階的に追加するため、v2.0 では**重量・レップ数の記録と閲覧**に集中する。

※実装ルールは `.claude/rules/` を参照すること。

### 1.1 v1.x からの変更点サマリー

| カテゴリ         | 内容                                                                                   |
| ---------------- | -------------------------------------------------------------------------------------- |
| DB               | `exercises` / `workout_sessions` / `workout_sets` テーブル追加                          |
| 画面             | トップナビゲーション追加。「ホーム」「トレーニング」の2タブ構成に拡張                 |
| ルーティング     | `/workouts`（一覧）/ `/workouts/new`（新規作成）/ `/workouts/[sessionId]`（編集）を追加 |
| Server Actions   | トレーニング記録用の Server Actions を追加                                              |
| シードデータ     | 代表的な種目（スクワット・ベンチプレス等）をグローバルマスタとして事前投入              |

## 3. DB設計

### 3.1 追加テーブル

#### exercises テーブル（種目マスタ）

| カラム      | 型          | 制約                                   |
| ----------- | ----------- | -------------------------------------- |
| id          | uuid        | PRIMARY KEY, DEFAULT gen_random_uuid() |
| user_id     | uuid        | NULL許容, REFERENCES users(id) ON DELETE CASCADE |
| name        | text        | NOT NULL                               |
| body_part   | text        | NOT NULL（下記 enum）                  |
| is_default  | boolean     | NOT NULL, DEFAULT false                |
| created_at  | timestamptz | DEFAULT now()                          |
| updated_at  | timestamptz | DEFAULT now()                          |

**制約:**
- `UNIQUE (user_id, name)` — 同一ユーザー内での重複防止（NULL は別扱い）
- `body_part` チェック制約: `'chest' | 'back' | 'shoulder' | 'arm' | 'leg' | 'abs'`

**設計方針:**
- `user_id` が `NULL` のレコードは**全ユーザー共通のグローバルマスタ**（シードで投入）
- `user_id` が設定されたレコードはそのユーザーのカスタム種目
- UI では両方を混ぜて表示し、カスタム種目はユーザー自身のみ編集・削除可能
- `is_default = true` はシードでのプリセット種目を示すフラグ（削除不可判定用）

#### workout_sessions テーブル（セッション）

| カラム       | 型          | 制約                                                 |
| ------------ | ----------- | ---------------------------------------------------- |
| id           | uuid        | PRIMARY KEY, DEFAULT gen_random_uuid()               |
| user_id      | uuid        | NOT NULL, REFERENCES users(id) ON DELETE CASCADE     |
| date         | date        | NOT NULL                                             |
| created_at   | timestamptz | DEFAULT now()                                        |
| updated_at   | timestamptz | DEFAULT now()                                        |

**制約:**
- `body_records` と異なり **`UNIQUE (user_id, date)` は設定しない**（1日複数セッション可）

#### workout_sets テーブル（セット記録）

| カラム      | 型            | 制約                                                     |
| ----------- | ------------- | -------------------------------------------------------- |
| id          | uuid          | PRIMARY KEY, DEFAULT gen_random_uuid()                   |
| user_id     | uuid          | NOT NULL, REFERENCES users(id) ON DELETE CASCADE         |
| session_id  | uuid          | NOT NULL, REFERENCES workout_sessions(id) ON DELETE CASCADE |
| exercise_id | uuid          | NOT NULL, REFERENCES exercises(id) ON DELETE RESTRICT    |
| set_number  | integer       | NOT NULL, CHECK (set_number >= 1)                        |
| weight_kg   | numeric(5,2)  | NOT NULL, CHECK (weight_kg >= 0)                         |
| reps        | integer       | NOT NULL, CHECK (reps >= 1)                              |
| memo        | text          | 任意                                                     |
| created_at  | timestamptz   | DEFAULT now()                                            |
| updated_at  | timestamptz   | DEFAULT now()                                            |

**制約:**
- `UNIQUE (session_id, exercise_id, set_number)` — 同一セッション・同一種目内でのセット番号重複防止

**設計方針:**
- `user_id` を正規化せず保持する理由: v1.1 の RLS パターンと一貫性を取り、ポリシーをシンプルに保つ（JOIN なし）
- `exercise_id` は `ON DELETE RESTRICT`: セット記録に紐づく種目は削除を禁止（誤削除防止）

### 3.2 RLS ポリシー

v1.1 と同様に、全テーブルで RLS を有効化する。

```sql
-- exercises: グローバルマスタ（user_id IS NULL）は全ユーザー閲覧可、自分のカスタムは自分のみ
ALTER TABLE exercises ENABLE ROW LEVEL SECURITY;

CREATE POLICY exercises_select_policy ON exercises
  FOR SELECT
  USING (
    user_id IS NULL
    OR user_id = current_setting('app.current_user_id', true)::uuid
  );

CREATE POLICY exercises_write_policy ON exercises
  FOR ALL
  USING (user_id = current_setting('app.current_user_id', true)::uuid)
  WITH CHECK (user_id = current_setting('app.current_user_id', true)::uuid);

-- workout_sessions / workout_sets: 自分のデータのみ
ALTER TABLE workout_sessions ENABLE ROW LEVEL SECURITY;
ALTER TABLE workout_sets ENABLE ROW LEVEL SECURITY;

CREATE POLICY workout_sessions_user_policy ON workout_sessions
  FOR ALL
  USING (user_id = current_setting('app.current_user_id', true)::uuid);

CREATE POLICY workout_sets_user_policy ON workout_sets
  FOR ALL
  USING (user_id = current_setting('app.current_user_id', true)::uuid);
```

### 4.4 セッション新規作成ページ（`/workouts/new`）

1画面でセッション全体を入力する構成。入力内容は自動的に下書き保存される（5.8 参照）。

**セクション1: セッション情報**

| 要素 | 詳細                           |
| ---- | ------------------------------ |
| 日付 | 今日の日付を表示（変更不可）   |

**セクション2: 種目・セット入力**

- 「+ 種目を追加」ボタン押下で種目選択モーダル（プリセット + カスタム一覧、検索フィルタ付き）
- 追加した種目ごとに、セット入力行を展開
  ```
  ■ ベンチプレス                                   [種目を削除]
    セット1:  [60] kg  × [10] 回  [メモ: 任意]    [削除]
    セット2:  [60] kg  × [ 8] 回  [メモ: 任意]    [削除]
    [+ セットを追加]
  ```
- セット番号は自動採番
- カスタム種目作成UIは種目選択モーダル内から遷移

**セクション3: 完了**

- 「完了」ボタン押下で Server Action を実行（`createWorkoutSessionAction`）
- バリデーション失敗時は画面内にインラインで表示（フォームは維持）
- 成功時は `/workouts/[sessionId]` へ遷移し、localStorage の下書きを削除する

## 5. 機能仕様

### 5.1 セッション登録（Create）

- `/workouts/new` から入力
- 1リクエストで「セッション + 種目グルーピング + 全セット」をトランザクションで保存
- **Server Actions は1つに集約**（`.claude/rules/04_data-mutation.rules.md` の方針）
- 成功後: セッション編集ページへ遷移し、`revalidateTag('workout-sessions')` でキャッシュ再検証

### 5.2 セッション編集（Update）

- `/workouts/[sessionId]` で編集
- 差分更新ではなく、**クライアント側のフォーム状態をまるごと送信 → サーバー側でセット・種目を総入れ替え**する戦略を採用
  - 理由: セット追加・削除・並び替えが多数発生するため、差分計算するより全入れ替えのほうがロジックが単純で整合性も取りやすい
  - セッション本体（`workout_sessions`）は UPDATE、セット（`workout_sets`）は DELETE + INSERT でトランザクション内実行
- 成功後: 同ページを再レンダリング（`revalidateTag`）

### 5.3 セッション削除（Delete）

- 編集ページの削除ボタン押下 → 確認モーダル → Server Action
- `workout_sets` は `ON DELETE CASCADE` で自動削除
- 成功後: `/workouts` へ遷移

### 5.4 セット単位の操作

セッション編集ページ内のみで完結する（セット単独のエンドポイントは設けない）。

- セットの追加・編集・削除はクライアント側のフォーム状態で管理（`useState` / `useReducer`）
- 保存ボタン押下で 5.2 の一括更新に集約

**なぜ1セットずつ Server Action を呼ばないのか:**
- 連続操作時の ラウンドトリップが増え UX が悪化する
- 整合性担保のためには最終的に一括保存のほうが確実
- `.claude/rules/04_data-mutation.rules.md` の「Server Actions は基本1つに集約」に整合

### 5.5 種目マスタの操作

| 操作             | 内容                                                                 |
| ---------------- | -------------------------------------------------------------------- |
| 一覧取得         | `SELECT * FROM exercises` （RLS により自分の + グローバル取得）       |
| カスタム種目追加 | `INSERT INTO exercises` with `user_id = <自分>`, `is_default = false` |
| カスタム種目編集 | `UPDATE` 自分のカスタム種目のみ。プリセットは不可                     |
| カスタム種目削除 | `DELETE` 自分のカスタム種目のみ。ただし `workout_sets` で参照中は `ON DELETE RESTRICT` により失敗 → UI で「使用中のため削除できません」を表示 |

### 5.6 データ取得パターン

`.claude/rules/03_data-fetching.rules.md` に従う。

| 画面                                 | パターン                                                                  |
| ------------------------------------ | ------------------------------------------------------------------------- |
| セッション一覧（`/workouts`）        | Server Component + fetch（Data Cache OFF、`revalidateTag` で再検証）       |
| セッション編集（`/workouts/[id]`）   | Server Component + fetch（Data Cache OFF）                                 |
| 種目選択モーダル（検索・インタラクティブ）| Client Component + TanStack Query + Route Handler（ユーザー操作トリガー） |

- 種目マスタはユーザー操作で頻繁にフィルタするため、TanStack Query で管理しクライアントサイドでキャッシュする
- セッションデータはページ遷移単位でSSRし、Data Cache は使わず `revalidateTag` のみで整合を取る

### 5.7 楽観的更新

v2.0 では**採用しない**。
- トレーニング記録は一括保存が中心で、即時反映が必要な単発アクションがない
- v2.1 以降で RPE 入力など細かい更新が入る場合に導入検討

### 5.8 下書き自動保存

新規作成ページ（`/workouts/new`）のフォーム状態を localStorage に自動保存する。

| 項目           | 内容                                                                 |
| -------------- | -------------------------------------------------------------------- |
| 保存先         | `localStorage`、キー: `workout-draft`                                |
| 保存タイミング | 種目・セットの追加・削除・入力変更のたびに即時書き込み               |
| データ形式     | `{ date: 'YYYY-MM-DD', exercises: ExerciseDraft[] }`                 |
| 復元条件       | ページマウント時に `date` が今日と一致する場合のみ復元する           |
| クリア条件     | 「完了」ボタン押下で保存成功後に削除する                             |

**再開バナー（`/workouts` 一覧ページ）**

- ページマウント時に localStorage を確認し、当日の下書きがあればバナーを表示
- 「再開」→ `/workouts/new` へ遷移（ページマウント時に自動復元）
- 「破棄」→ `localStorage.removeItem('workout-draft')` してバナーを非表示

**設計方針**

- 下書きは常に「作業中1件分」のみ保持する（複数管理は不要）
- 別日付の下書きは復元せず無視する（今日のトレーニングのみ対象）
- Apple Health / Strong 等の主要トレーニングアプリが採用するパターン

## 6. バリデーション仕様

zod スキーマで Server Actions 側で検証する。

### 6.1 ExerciseSchema（カスタム種目）

| フィールド | ルール                                                                  |
| ---------- | ----------------------------------------------------------------------- |
| name       | 必須。1〜50文字                                                         |
| body_part  | 必須。enum `'chest' \| 'back' \| 'shoulder' \| 'arm' \| 'leg' \| 'abs'` |

### 6.2 WorkoutSessionSchema

| フィールド | ルール                                                        |
| ---------- | ------------------------------------------------------------- |
| date       | 必須。YYYY-MM-DD。未来日不可                                  |
| sets       | 必須。1件以上のセット配列（下記 WorkoutSetSchema の配列）     |

### 6.3 WorkoutSetSchema

| フィールド  | ルール                                              |
| ----------- | --------------------------------------------------- |
| exercise_id | 必須。uuid                                          |
| set_number  | 必須。整数、1以上                                   |
| weight_kg   | 必須。数値、0〜1000、小数点2桁                      |
| reps        | 必須。整数、1〜1000                                 |
| memo        | 任意。0〜200文字                                    |

### 6.4 エッジケースのバリデーション

- セッション保存時、`sets` 配列の中で `(exercise_id, set_number)` の組み合わせが重複していないことをサーバー側で検証
- 同一セッション内に0セットの種目が残らないよう、保存前にフロントでフィルタ

## 8. 環境変数・セキュリティ

v1.1 から変更なし。既存の `DATABASE_URL` / `AUTH_*` をそのまま利用。

セキュリティ方針:
- トレーニングデータも RLS により他ユーザーからのアクセスを DB 層で遮断
- Server Actions で必ずセッションから `user_id` を取得し、全クエリに付与

## 9. 型定義

### BodyPart

```ts
type BodyPart = 'chest' | 'back' | 'shoulder' | 'arm' | 'leg' | 'abs';
```

### Exercise

```ts
type Exercise = {
  id: string;
```

## 10. マイグレーション方針

v1.1 からの DB 移行手順。

1. `db/migration_v2.0.sql` に以下を記述し実行:
   - `exercises` / `workout_sessions` / `workout_sets` テーブル作成
   - 各テーブルに RLS ポリシー設定
2. `db/seed_v2.0.sql` でプリセット種目を投入
3. 既存の `body_records` / `goals` には影響なし（変更なし）

**ロールバック:**

- 失敗時は新規追加した3テーブルを `DROP TABLE ... CASCADE` で削除するのみ（既存データへの影響なし）
