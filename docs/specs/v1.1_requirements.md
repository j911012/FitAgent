# FitLog

**v1.1 要件仕様書**

認証・ユーザー分離

2026年4月 / Version 1.1

---

## 1. 概要

v1.0で構築した体重・体脂肪率CRUD基盤に、認証とユーザー分離を追加する。Auth.js（NextAuth v5）でGoogle OAuthを導入し、ユーザーごとにデータを分離する。Neon Row Level Security（RLS）でDB層でもデータ保護を担保する。
※実際の実装ルールなどはrulesを参照すること

### 1.1 v1.0からの変更点サマリー

| カテゴリ     | 内容                                                                          |
| ------------ | ----------------------------------------------------------------------------- |
| 認証         | Auth.js（NextAuth v5）+ Google OAuth                                          |
| DB           | `users` / `accounts` テーブル追加。`body_records` / `goals` に `user_id` 追加 |
| セキュリティ | Neon RLS有効化。環境変数整理                                                  |
| 画面         | ログインページ追加。ヘッダーにユーザー情報・ログアウト追加                    |
| ルーティング | `(public)` グループにログインページ配置。`(private)` グループで認証ガード     |

| -------------- | ----------- | -------------------------------------- |

| email          | text        | UNIQUE, 任意                           |
| email_verified | timestamptz | 任意                                   |
| image          | text        | 任意（Googleプロフィール画像URL）      |
| created_at     | timestamptz | DEFAULT now()                          |
| updated_at     | timestamptz | DEFAULT now()                          |

#### accounts テーブル（Auth.js Neon Adapterが自動管理）

| カラム              | 型   | 制約                                   |
| ------------------- | ---- | -------------------------------------- |
| id                  | uuid | PRIMARY KEY, DEFAULT gen_random_uuid() |
| user_id             | uuid | REFERENCES users(id) ON DELETE CASCADE |

| provider            | text | NOT NULL                               |

| access_token        | text | 任意                                   |

| scope               | text | 任意                                   |

### 3.2 変更テーブル

#### body_records テーブル（`user_id` カラム追加）

| カラム      | 型           | 制約                                                 |
| ----------- | ------------ | ---------------------------------------------------- |
| id          | uuid         | PRIMARY KEY, DEFAULT gen_random_uuid()               |
| **user_id** | **uuid**     | **NOT NULL, REFERENCES users(id) ON DELETE CASCADE** |
| date        | date         | NOT NULL                                             |
| weight_kg   | numeric(4,1) | NOT NULL                                             |
| body_fat    | numeric(4,1) | 任意 (NULL許容)                                      |
| created_at  | timestamptz  | DEFAULT now()                                        |
| updated_at  | timestamptz  | DEFAULT now()                                        |

> **制約変更:** `UNIQUE(date)` → `UNIQUE(user_id, date)` に変更（同一ユーザーの同日重複を防ぐ）

#### goals テーブル（`user_id` カラム追加）

| カラム           | 型           | 制約                                                         |
| ---------------- | ------------ | ------------------------------------------------------------ |
| id               | uuid         | PRIMARY KEY, DEFAULT gen_random_uuid()                       |
| **user_id**      | **uuid**     | **NOT NULL, UNIQUE, REFERENCES users(id) ON DELETE CASCADE** |
| target_weight_kg | numeric(4,1) | 任意                                                         |
| target_body_fat  | numeric(4,1) | 任意                                                         |
| created_at       | timestamptz  | DEFAULT now()                                                |
| updated_at       | timestamptz  | DEFAULT now()                                                |

> **制約追加:** `user_id` に `UNIQUE` 制約（1ユーザー1レコード）

### 3.3 Neon Row Level Security（RLS）

全テーブルにRLSを有効化し、アプリケーション層とは独立したDB層のアクセス制御を実現する。

```sql
-- body_records RLS
ALTER TABLE body_records ENABLE ROW LEVEL SECURITY;

CREATE POLICY "ユーザーは自分のレコードのみ操作可能"
  ON body_records
  FOR ALL
  USING (user_id = current_setting('app.current_user_id')::uuid);

-- goals RLS
ALTER TABLE goals ENABLE ROW LEVEL SECURITY;

CREATE POLICY "ユーザーは自分の目標のみ操作可能"
  ON goals
  FOR ALL
  USING (user_id = current_setting('app.current_user_id')::uuid);
```

> **設定方法:** 各DBリクエストの前に `SET app.current_user_id = '<user_id>'` を実行する。Server ActionsとRoute Handlersの共通リクエスト関数で設定する。

## 5. 機能仕様

### 5.1 ログイン

- `/login` のGoogleログインボタン押下でGoogle OAuthフローを開始
- Google認証成功後、`/` にリダイレクト
- 初回ログイン時、`users` / `accounts` テーブルにレコードを自動作成（Auth.js Adapterが処理）

### 5.2 ログアウト

- ヘッダーのログアウトボタン押下でセッションを破棄
- `/login` にリダイレクト

### 5.3 認証ガード（Middleware）

- `(private)` 配下の全ルートに認証チェックを適用
- 未認証ユーザーは `/login` にリダイレクト
- 認証済みユーザーが `/login` にアクセスした場合は `/` にリダイレクト

### 5.4 ユーザー別データ分離

- Server Actions / API Route Handlers でセッションから `user_id` を取得
- 全DBクエリに `WHERE user_id = :userId` を付与
- Neon RLSを二重の安全網として機能させる

### 5.5 CRUD操作（v1.0から変更点のみ）

| 操作           | v1.0           | v1.1の変更                                    |
| -------------- | -------------- | --------------------------------------------- |
| 記録（Create） | 全ユーザー共通 | ログインユーザーの `user_id` をセットして保存 |
| 編集（Update） | IDで操作       | IDに加え `user_id` で自分のレコードのみ更新   |
| 削除（Delete） | IDで操作       | IDに加え `user_id` で自分のレコードのみ削除   |
| 目標設定       | 全ユーザー共通 | ログインユーザーの目標のみ取得・更新          |

## 6. バリデーション仕様

v1.0の入力バリデーション（体重・体脂肪率・目標値）はそのまま継承。認証関連の追加バリデーションはなし（Auth.jsとGoogleが処理）。

### 認証エラーのハンドリング

| ケース             | 対応                                                |
| ------------------ | --------------------------------------------------- |
| Googleログイン拒否 | `/login?error=OAuthAccountNotLinked` にリダイレクト |
| セッション切れ     | Middlewareが `/login` にリダイレクト                |
| 不正なCSRFトークン | Auth.jsが自動処理                                   |

## 8. 環境変数・セキュリティ

| 変数名                | 用途                                          | スコープ     |
| --------------------- | --------------------------------------------- | ------------ |
| DATABASE_URL          | NeonのPostgreSQL接続文字列（Pooled）          | サーバーのみ |
| DATABASE_URL_UNPOOLED | マイグレーション用接続文字列                  | サーバーのみ |
| AUTH_SECRET           | セッションJWT署名用シークレット（32文字以上） | サーバーのみ |
| AUTH_GOOGLE_ID        | Google OAuth クライアントID                   | サーバーのみ |
| AUTH_GOOGLE_SECRET    | Google OAuth クライアントシークレット         | サーバーのみ |
| NEXTAUTH_URL          | アプリのベースURL（本番環境）                 | サーバーのみ |

### セキュリティ方針

- `AUTH_SECRET` は `openssl rand -base64 32` で生成し、`.env.local` に記載
- Google Cloud ConsoleでOAuth同意画面を設定し、許可するリダイレクトURIを限定
- Vercelデプロイ時は環境変数をVercelダッシュボードで管理（`.env.local` は本番に使わない）
- RLSにより、アプリケーションバグによる他ユーザーデータへのアクセスをDB層でも防止

## 10. マイグレーション方針

v1.0からv1.1へのDB移行手順。

1. `users` / `accounts` テーブルを新規作成（Auth.jsマイグレーションスクリプト実行）
2. `body_records` に `user_id` カラムを追加（NOT NULL制約はマイグレーション後に付与）
3. `goals` に `user_id` カラムを追加（同上）
4. v1.0の既存データは開発用のみのため、マイグレーション時に一括削除して問題なし
5. UNIQUE制約を `(date)` から `(user_id, date)` に変更
6. RLSポリシーを設定

> **v1.0の既存レコードについて:** 開発環境のみに存在し、本番リリース前のため既存データを引き継ぐ必要なし。本番リリース後にユーザーが増えてから移行が必要になるケースはv1.2以降で対応する。

## 12. 実装順序

1. **Auth.js セットアップ**
   - `next-auth` / `@auth/neon-adapter` インストール
   - `auth.ts` 設定ファイル作成（Google Provider）
   - `app/api/auth/[...nextauth]/route.ts` 作成

2. **DB マイグレーション**
   - Auth.js用テーブル（`users` / `accounts`）作成
   - `body_records` / `goals` に `user_id` カラム追加
   - UNIQUE制約変更
   - RLSポリシー設定

3. **Middleware 実装**
   - 認証ガードの設定（`(private)` 配下を保護）
   - ログイン済みユーザーの `/login` アクセスをリダイレクト

4. **ログインページ実装**
   - `/login` ページ作成
   - Googleログインボタン

5. **ヘッダーコンポーネント実装**
   - ユーザーアバター・名前表示
   - ログアウトボタン

6. **Server Actions 更新**
   - セッションから `user_id` 取得
   - 全クエリに `user_id` フィルタ追加
   - RLS用 `app.current_user_id` 設定
